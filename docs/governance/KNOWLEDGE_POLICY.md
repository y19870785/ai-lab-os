# AI-Lab Knowledge Policy

> 知识管理规范。定义知识的分类、进入、审核、版本、删除和可信度管理。

---

## 1. 知识分类

AI-Lab 的五种知识类型各有明确的定义和准入标准：

| 知识类型 | 定义 | 准入标准 |
| --- | --- | --- |
| **Document Knowledge** | 外部文档数据（PDF / Markdown / HTML / CSV） | 文档需有明确的来源和标题 |
| **Entity Knowledge** | 命名实体及其属性（人 / 公司 / 概念 / 产品） | 需来自可信来源或有足够的引用 |
| **Relation Knowledge** | 实体间语义关系 | 需有明确的 relation_type 和置信度标注 |
| **Experience Knowledge** | Best Practice、方法论、工作流模板 | 需有明确的适用场景和步骤描述 |
| **Decision Knowledge** | 历史决策记录及推理过程 | 需记录所有备选方案和选择理由 |

### 分类原则

- 一条知识**只能属于一个类型**，不可交叉归类
- 如果一条知识同时满足多个类型，选择信息量最丰富的那一个
- 不确定的归类 → 先归为 Document Knowledge，后续通过 Entity Extractor 升级

## 2. 知识进入流程

```
新知识来源
    │
    ▼
[1] 来源识别
    │  文件 / URL / API / 手动输入 / Memory 升级建议
    ▼
[2] 格式解析
    │  按 SourceType 路由到对应 Parser
    │  提取元数据和原始内容
    ▼
[3] 分类判定
    │  确定 KnowledgeType
    │  不确定 → 默认为 Document Knowledge
    ▼
[4] 预处理
    │  Chunk（文档知识）
    │  实体识别（可选）
    │  Embedding 向量化
    ▼
[5] 审核（可选）
    │  高风险/高权限知识需要人工审核
    │  低风险知识自动进入
    ▼
[6] 存储
    │  写入对应 Store（Document / Entity / Graph / Vector）
    │  发布 KnowledgeEvent
    ▼
完成
```

### 自动进入 vs 人工审核

| 知识来源 | 自动进入 | 人工审核 |
| --- | --- | --- |
| 通过 Ingestion API 批量导入的文档 | ✅ 自动 | 可选 |
| 用户手动输入的知识条目 | ✅ 自动 | — |
| Agent 对话中自动提取的实体 | ❌ 待确认 | ✅ 需要 |
| Memory Layer 升级建议 | ❌ 待确认 | ✅ 需要 |
| 系统配置文件/文档 | ✅ 自动 | 可选 |

## 3. 知识审计

### 需要审计的操作

| 操作 | 审计级别 | 记录内容 |
| --- | --- | --- |
| 知识新增 | BASIC | 操作者、知识类型、来源 |
| 知识更新 | BASIC | 操作者、更新字段 |
| 知识删除 | FULL | 操作者、删除原因、备份位置 |
| 知识导出 | FULL | 操作者、导出范围、用途 |
| 权限变更 | FULL | 操作者、变更前后对比 |

### 审计日志存储

- 审计日志存储在 `logs/audit/` 目录，按日期分文件
- 格式：JSON Lines（每行一条审计记录）
- 保留期：至少 180 天

## 4. 知识版本管理

| 规则 | 说明 |
| --- | --- |
| 版本号 | 每次更新自动递增 version 字段 |
| 回滚 | 保留最近 5 个版本的历史记录 |
| 冲突 | 同一知识条目的并发更新，后写者覆盖（last-write-wins）|
| 引用 | 其他知识条目引用时使用 ID，不绑定版本号 |

### 版本升级建议

- 内容修正（typo、错误数据）→ version +1
- 内容扩展（新增章节、补充论据）→ version +1
- 结构重构（改了 chunk 策略、重新 embedding）→ version 重置为 1，标记为结构变更

## 5. 知识删除规则

| 场景 | 处理方式 | 数据保留 |
| --- | --- | --- |
| 用户手动删除 | 软删除（标记 deleted_at）| 保留 30 天后物理清除 |
| 来源文档被删除 | 级联删除关联的 chunks 和 embedding | 同上 |
| 去重发现 | 合并到保留条目，删除重复 | 立即物理清除 |
| 存储空间不足 | 按 access_level + 最后访问时间淘汰 | 回收前通知 owner |

**软删除 vs 物理删除：**
- 软删除：`deleted_at` 字段非空，知识不可检索但保留在数据库
- 物理删除：从数据库和向量存储中彻底移除，不可恢复

## 6. 知识可信度管理

每类知识都有自己的置信度机制：

| 知识类型 | 置信度字段 | 评分方式 | 自动调整 |
| --- | --- | --- | --- |
| Document | 无（来源可信度） | 由 SourceType 决定 | 固定 |
| Entity | confidence: float | 由提取模型 + 人工确认决定 | 引用越多越高 |
| Relation | confidence: float | 由提取模型 + 来源数量决定 | 多源验证后提升 |
| Experience | confidence: float | 由创建者设定 + 使用反馈调整 | 成功引用 +1，失败引用 -1 |
| Decision | 无（记录性知识） | 不评分 | N/A |

### 置信度含义

| 置信度 | 含义 |
| --- | --- |
| 1.0 | 已验证：经过人工确认或多源交叉验证 |
| 0.7 - 0.9 | 高可信：单一可信来源或自动提取置信度高 |
| 0.4 - 0.6 | 中可信：自动提取或单次交互中提及 |
| 0.1 - 0.3 | 低可信：推测性、未验证 |
| 0.0 | 不可信（仅用于消歧，标记为 "resolved_false"） |
