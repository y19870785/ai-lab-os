# AI-Lab · Codex 工程规范

## 每轮交付前强制检查

以下三项检查必须在交付报告之前完成，禁止跳过。

### Gate 1：文件清单验证

交付前，对每个声称新增或修改的文件，执行存在性检查：

```
Get-ChildItem -Recurse -Name | Select-String "<文件名>"
```

如果返回空，说明文件未实际创建。**不允许在自检失败的情况下输出交付报告。**
本轮新增和修改的每一个文件都必须在此检查中确认存在。

### Gate 2：入口可执行性验证

如果本轮修改了 CLI / 启动脚本 / API / 交互入口，必须实际执行一次：

- 非交互入口：检查退出码（exit code 必须为 0）
- 交互入口：检查输出包含预期文本（如版本号、欢迎信息）

**禁止在未实际运行的情况下编造交互输出、截图或示例对话。**
如果配置了真实 API Key，至少完成一轮真实 LLM 调用验证。

### Gate 3：测试回归

```
python -m pytest tests -q --tb=no
```

- 通过数必须 ≥ 修改前的通过数
- 0 新增失败（网络超时导致的 real 测试 error 除外，但必须注释说明）
- 不得为通过测试而降低覆盖率、skip 旧测试、或放宽断言

### Gate 4（新增文件专用）：写入后立即回读

每写完一个文件，立刻用 `Get-Content` 读回前 5 行，确认文件非空、编码正确、内容匹配预期。
禁止批量写文件后一次性声称「已创建」。

---

## 开发原则

1. 先侦察，后动手。修改前必须读当前文件内容。
2. 一步一验证。写完就跑测试，不要攒到最后。
3. 不编造输出。所有运行结果来自实际执行。
4. 保持文档同步。CHANGELOG、ARCHITECTURE、README 每次更新。
5. 中文文档优先。README、CHANGELOG、ARCHITECTURE 全部用中文。

---

## 禁止事项

- 禁止在未实际执行的情况下输出「实测通过」「交互成功」等验收结论
- 禁止批量声称创建文件而不逐一验证
- 禁止为通过测试而降低覆盖率或放宽断言
- 禁止跳过 Gate 1-4 直接输出交付报告
